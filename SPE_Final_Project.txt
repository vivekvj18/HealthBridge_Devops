1 System Overview
MovieTime is a full-stack, microservices-driven online movie booking platform designed to deliver a
seamless digital ticketing experience. The system enables end users to create accounts, authenticate
securely, browse movies, explore available showtimes, reserve seats, complete online payments, and receive
booking confirmations through email notifications. The architecture follows modern distributed system
principles, ensuring scalability, reliability, and maintainability across all operational environments.
The platform is organized into three primary layers, each responsible for a distinct operational domain:
• Frontend Layer: A React-based web application providing the user interface for customers and
administrative users.
• Backend Layer: A collection of independent Spring Boot microservices implementing business
logic and handling domain responsibilities.
• Infrastructure and DevOps Layer: Deployment and operational tooling built on Docker,
Kubernetes, Jenkins, and the ELK stack for automation, scalability, and observability.

1.1 User Roles
MovieTime supports multiple interacting roles within the system. End users engage with the platform to
browse content and make bookings, while administrators manage the catalogue of movies and showtimes.
In addition, several internal system services operate autonomously to enable inter-service communication,
monitoring, and orchestration.
• User: Performs registration, authentication, movie exploration, seat booking, completing payments,
and receiving transactional notifications.
• Admin: Manages movies, show schedules, and system-level entities through privileged interfaces.
• System Services: Microservices coordinating with each other through REST APIs and asynchronous message queues.

1.2 Core Functionalities
The platform provides a wide range of functional capabilities essential for a production-ready booking
system. These include secure access control using JWT, robust booking workflows with seat availability
enforcement, integrated payment handling, and automated email-based communication. Centralized
logging and service discovery further enhance operational resilience.
Key capabilities include:
• Secure user authentication and authorization
• Movie and show management
• Real-time ticket booking and seat availability tracking
• Dedicated payment processing service
• Automated email notifications
• Dynamic routing and service discovery
• Centralized logging and monitoring

6

1.3 Microservices-Based Design
The backend architecture is fully microservices-oriented. Each business capability is encapsulated within
its own service, deployed independently, and backed by a dedicated database. This separation enables
fault isolation, horizontal scalability, and flexibility for future extensions using heterogeneous technologies.
The primary microservices include:
• User Service
• Movie Service
• Booking Service
• Payment Service
• Notification Service
• API Gateway
• Eureka Discovery Server

Figure 1: High-Level Request and Processing Flow
A typical request proceeds as follows:
1. The frontend application sends all API calls to the API Gateway.
2. The gateway performs routing to the appropriate backend service.
3. Microservices communicate with each other through REST APIs or RabbitMQ events.
4. Data persistence is handled through service-specific PostgreSQL databases.
5. The processed response is returned to the frontend through the gateway.
6. All logs generated by the microservices and infrastructure are forwarded to the ELK stack.

20 Testing Strategy
The MovieTime platform uses a layered testing strategy to ensure correctness and reliability
across all microservices. Each test type is designed to validate a specific aspect of the system

independently. They use JUnit, Spring Boot Test, and Mockito to mock external dependencies

A dedicated test profile disables Eureka, messaging, SMTP, and external REST calls, ensuring

20.3 Integration Testing
Integration tests validate interactions between controllers, services, and repositories within a
microservice. Spring Boot automatically loads a test application context to simulate realistic
service behavior without deploying the system.

20.4 Message Queue Testing
Messaging logic is tested using mocked RabbitMQ components to verify event publishing, consumption, and serialization. This ensures correctness in the Booking, Payment, and Notification
workflows without requiring a live message broker.

20.5 Security Testing

requests.

20.6 CI-Based Automated Testing
Jenkins runs all tests automatically during the CI pipeline and blocks deployments if any
failures occur. This guarantees that only stable, verified code proceeds to containerization and
deployment.

20.7 Resilience of the Testing Approach
The strategy ensures that each microservice remains fully testable even if infrastructure components are offline. This isolation increases confidence in application correctness and supports
rapid development cycles.

23 Why Ansible Is Not Required
Configuration management tools such as Ansible are commonly used for provisioning servers, automating deployments, and maintaining system configurations across remote machines. However,
in this project, Kubernetes already provides all the orchestration, automation, and infrastructure
management capabilities required by the backend microservices.
Since the entire platform runs inside a Kubernetes cluster, many responsibilities traditionally
handled by Ansible are fully replaced by Kubernetes features such as:
• Declarative infrastructure management using YAML manifests
• Automated container deployment and rollout strategies
• Built-in service discovery and networking
• Scaling through replicas and horizontal pod autoscaling
• Secret and configuration management through ConfigMaps and Secrets
Additionally, Minikube is used as a local simulation of a remote cluster. Introducing Ansible
would create an unnecessary additional layer, akin to running a virtual machine inside another
virtual machine. It would increase operational complexity without offering meaningful advantages.
